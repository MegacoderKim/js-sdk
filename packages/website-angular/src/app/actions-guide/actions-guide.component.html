Actions guide

Introduction

Action is the unit of tracking with HyperTrack. It is the starting point for building your use case.

This guide covers all that you need to know about actions in order to use them effectively. We will also cover common use cases that developers have built with HyperTrack and point you to quick APIs and tutorials to build them with actions.

Feel free to build anything that you can imagine with actions that we might not have imagined yet. We would love to hear from you when you do.

1. Use model

As a first step, you need to decide when to create and complete an action.

This is an important decision and we would encourage you to be thoughtful about it. We would be happy to chat with you about this to help.

Creating actions will start tracking the movement of the user and completing the action will end it.

When multiple actions are created for a user, tracking starts with the first action created and ends with the last action completed. You may create as many actions as you want for a user at any point in time.

1.1. Ways to create actions

Actions may be created:

  1. When actions are assigned to the user by your system in the back-end
  2. When actions are started by the user in your app
  3. At a set time, optionally with a recurring schedule
  4. At a fixed duration before it is expected to complete
  5. Upon entering/exiting a place, optionally from a list of places

1.2. Ways to complete actions

Actions may be completed:

  1. When actions are completed by the user in your app
  2. When actions are completed by your system in the back-end
  3. Immediately upon creation
  4. At a set time
  5. At a fixed duration after it is created or expected to complete
  6. Upon entering/exiting a place

1.3. Action types

Each action may be created with an action type picked from our list or a custom type as defined by you. Actions for multiple types may be created for a user.

1.4. Defining your own actions

Most developers create actions from the back-end or app when the respective action is created or assigned to a user in their own system. They complete actions from the app or back-end when the equivalent action is completed by the user in their own system. Developers specify an expected place where the action is expected to complete in order to unlock additional features.

1.5. What you get with actions


Each action gets you:

  • Live tracking URL with live location, activity status and device health status
  • Tracking experience
  • Events
  • Location
  • Activity
  • Health
  • Distance
  • Placeline

Note that each action gives you unlimited:

  • Data tracked - time, distance, location points, number of events
  • Data consumed - as data, visuals, or events
  • Consumption channels - use cases, API gets, tracking views, consuming applications, webhooks received, Slack alerts or team members with dashboard access

The time and place for action creation and completion are available in the action API as well.


Additionally, when you specify expected place, you get:

  • Live tracking URL with live ETA, expected route and arrival status
  • Tracking experience with expected places
  • Delay events
  • ETA
  • Expected route
  • Remaining distance
  • Arrival status

For the purpose of computing ETAs, the system assumes that actions with expected places will be completed in the order in which they were created. If different, you may specify the expected sequence. ETAs for all created actions with expected place will update in real-time as the user goes about completing them.

For more detail on what you get with actions, see the what you GET section and tracking experience section.

2. Creating and completing actions

2.1. APIs to create, complete and cancel

API call URL Description
Create actions actions/ Create one or more actions
Complete an action actions/ <id>/complete/ Complete an action
Cancel an action actions/ <id>/cancel/ Cancel an action

2.2. Entity IDs and metadata from your system when creating actions

All parameters listed here except ID of user are OPTIONAL. See create actions API reference for the full list.

Parameters Type Description
user_id or user_unique_id String user_id is the ID of user generated by HyperTrack and user_unique_id is the ID of user provided by you. Use either one when creating action.
unique_id String Unique identifier for the action in your system
collection_id String Identifier for the collection of actions in your system. See below for more detail.
type String Type of action, e.g., meetup, pickup, delivery, dropoff, visit, sos, payment, stopover, task, or custom
metadata JSON Custom key-values pairs in your system that you want to associate with the action E.g. Customer - John Smith

2.2.1. Using collections

Use collections to group actions. Examples:

  • A group of friends meeting up and sharing live location with each other

  • An order that requires multiple actions to be tracked, such as a pickup followed by a dropoff

2.3. Rules to automatically create/complete actions

You can define rules to automatically create and complete actions at set time(s) or place(s). Rules to create or complete actions at places might be when a user arrives at or leaves from a list of places.

When you followed the SDK integration guide, you may have set the default rule to create/complete actions that track users through the day. This rule autocreates a new action everyday at midnight with type track_through_the_day and autocomplete_rule parameter set to type timefence so that it is completed at a fixed duration after it is created.

2.4. Pause and resume tracking

To protect your user’s privacy, you may pause and resume tracking even when there are actions created for the user that have not yet been completed.

2.5. Expected place and time

These are OPTIONAL parameters. See create actions API reference for more.

Parameters Type Description
expected_place JSON Object of place where the action is expected to be completed. Use it to get ETA, expected route, remaining miles and delay events.
expected_at ISO datetime string Time at which action is scheduled to complete. Will be used to compute delays.

3. What you GET

3.1. APIs to GET the action objects and its properties

API call URL Description
Get an action actions/ <id>/ Get an action object
Get multiple actions actions/ Lists all actions objects. Use parameters to filter and order

3.2. Properties returned in action object

Property Type Description
id UUID String Unique identifier for the action object created by HyperTrack when action was created
user JSON Object of user for whom action is created
unique_id String Unique identifier provided by you while creating the action
collection_id String Identifier for the collection of actions in your system provided by you while creating action.
metadata JSON Custom key-values pairs in your system that you want to associate with the action. This is provided while creating action.
type String Type of action
tracking_url String Live tracking URL with live location, activity status and device health status
location JSON Object with live location of action
activity JSON Object with live activity of action
health JSON Object with live device health
status String Current status of the action. See below for possible values
updated_at ISO datetime string Last time when the action object was updated by HyperTrack
created_at ISO datetime string Time when action was successfully created
completed_at ISO datetime string Time when action was successfully completed, autocompleted, canceled or autocanceled
started_place JSON Object of place where tracking started for action
completed_place JSON Object of place where action was completed
distance Integer Distance traveled from the previous action* completed, in meters.
shortest_distance Integer Shortest distance from the previous action*, in meters.

* Distance of the first action will be computed from the time/place tracking started. Tracking starts when one or more actions are created for the user and stops when all actions for the user are completed.

3.2.1. Status

The possible values for status are:

  • created: action created successfully

  • completed: action completed successfully

  • autocompleted: action automatically completed successfully due to autocomplete_rule parameter

  • canceled: action canceled successfully

  • autocanceled: action automatically canceled successfully due to autocancel_after parameter

3.3. Additional properties returned when action created with expected_place

Property Type Description
arrival_status String Current arrival status of the action. See below for possible values.
eta ISO datetime string Current ETA for user to get to expected_place on most favorable route factoring live traffic conditions
eta_at_creation ISO datetime string ETA when action was created. Use it to compare current eta with the original expectation set
expected_route String Latlongs for most favorable route for user to get to expected_place factoring live traffic conditions. Also the route based on which eta is computed.
remaining_distance Integer Distance remaining, in meters

3.3.1. Arrival status

The possible values for arrival_status are:

  • leaving_now: When user has moved <100 meter from the place where tracking started

  • on_the_way: When user has moved >100 meter from the place where tracking started

  • arriving: When user is <400 meter from expected_place

  • arrived: When user is <100 meter from expected_place

3.4. Default sequence of actions and APIs to modify

In case of multiple actions with expected_place created for a user, arrival_status, eta and other properties are computed for all actions with an expected_place. By default, the system expects actions to complete in the sequence they are created. The default sequence may be modified using the follow API calls. Note that actions with no expected_place are excluded from this computation and as such, do not have the additional properties that come with expected_place.

API call Method URL Description
Get sequence of actions with expected_place GET users/ <id>/sequence_actions/ Get the sequence in which actions with expected_place are expected to be completed. By default, they are expected to complete in the order they were created
Update sequence of actions with expected_place POST users/ <id>/sequence_actions/ Update the sequence in which actions with expected_place are expected to be completed. Use this to get better tracking experience, ETA, expected route, remaining miles and delay alerts for each action

3.5. APIs to GET data for common use cases

API URL Description Use cases
Mileage tracking actions/ <id>/mileage/ Get mileage for each action based on chosen parameters. By default, mileage is computed from the previous action completed. Mileage tracking
Placeline actions/placeline/ Gets placeline and location_time_series for action Order tracking Customer support Operations monitoring
Events actions/ <id>/events/ Get events for action Order tracking Customer support Operations monitoring

4. Build the tracking experience

There are multiple ways to build a live movement tracking experience for actions that are not yet completed.

  • Track via a web URL using the the unique trck.at tracking URL that is returned in the response object of the Action. This is a public URL that may be shared with customers or anyone you want.

  • Track via your HyperTrack dashboard or embed these views into your own dashboard.

  • Track via your native Android, iOS or JS web app
